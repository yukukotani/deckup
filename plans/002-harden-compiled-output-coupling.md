## Why this matters

Deckup rewrites Astro's _compiled_ JS output by searching for the exact string
`'$$renderComponent($$result, "Page", Page,'`. This marker is duplicated in
two places and assumes the Astro compiler's exact call formatting. At the same
time, the pnpm catalog pins `astro: latest`, so any Astro release that changes
compile output formatting (whitespace, argument spelling) silently breaks deck
builds with a cryptic count-mismatch error. This plan (a) pins the
Astro-family catalog entries to caret ranges so upgrades become deliberate,
(b) replaces the two duplicated exact-string markers with one shared,
whitespace-tolerant regex, and (c) leans on Plan 001's characterization tests
plus existing integration tests as the upgrade canary.

Marker usage 1 — span discovery:

```ts
// apps/cli/src/deckup-vite-plugins.ts:316-331
function findCompiledPagePropsSpans(source: string) {
  const marker = '$$renderComponent($$result, "Page", Page,';
  const spans: Array<{ start: number; end: number }> = [];
  let searchIndex = 0;
  while (searchIndex < source.length) {
    const markerIndex = source.indexOf(marker, searchIndex);
    if (markerIndex < 0) break;
    const start = source.indexOf("{", markerIndex + marker.length);
    if (start < 0) break;
    const end = findMatchingBrace(source, start);
    if (end < 0) break;
    spans.push({ start, end: end + 1 });
    searchIndex = end + 1;
  }
  return spans;
}
```

### Step 2: Extract a single whitespace-tolerant compiled-Page pattern

In `apps/cli/src/deckup-vite-plugins.ts`, add near the top (below the
existing `const resolvedVirtualDeckupThemeLayoutsId ...` declarations):

```ts
// Coupling point to Astro's compiled output. The Astro compiler emits
// `$$renderComponent($$result, "Page", Page, { ...props }, ...)` for each
// deck Page. If an Astro upgrade changes this shape, the integration tests
// in tests/astro.test.ts (buildDeck + data-deckup-layout assertions) and the
// characterization tests in tests/deckup-vite-plugins.test.ts fail first.
// Keep this tolerant to whitespace-only formatting changes.
const compiledPageRenderPattern = /\$\$renderComponent\(\s*\$\$result\s*,\s*"Page"\s*,\s*Page\s*,/g;

function findCompiledPageRenderMatches(source: string) {
  compiledPageRenderPattern.lastIndex = 0;
  return [...source.matchAll(compiledPageRenderPattern)];
}
```

Rewrite `findCompiledPagePropsSpans` (line 316) to use it:

```ts
function findCompiledPagePropsSpans(source: string) {
  const spans: Array<{ start: number; end: number }> = [];
  for (const match of findCompiledPageRenderMatches(source)) {
    const start = source.indexOf("{", match.index + match[0].length);
    if (start < 0) break;
    const end = findMatchingBrace(source, start);
    if (end < 0) break;
    spans.push({ start, end: end + 1 });
  }
  return spans;
}
```

Note the behavior difference vs. the old loop: the old code resumed searching
after `end + 1`; matchAll finds all marker occurrences up front. Since props
objects follow their marker and `findMatchingBrace` spans never start before
their own marker, overlapping is impossible for well-formed compiled output.
Plan 001's tests must still pass unchanged.

**Verify**: `vp run @deckup/cli#test` → exit 0, all Plan 001 characterization
tests pass unchanged.
